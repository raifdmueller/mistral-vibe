This Question Tree recovers the "theory" (Naur 1985) of the Mistral Vibe codebase
through recursive question refinement. Phase 1 builds the tree. Each leaf is
`[ANSWERED]` with `file:line` evidence or `[OPEN]` with a Category, an Ask role,
and why it cannot be answered from code.

The second level (Q1.1–Q5.5) is fixed. Q3.1–Q3.12 are the twelve arc42 chapters.
Q4.1–Q4.8 are the eight ISO/IEC 25010 characteristics; Q4.9 ranks them.

Scope: `vibe/` package, ~43.7k LOC Python, 228 test modules. Version 2.9.6
(`pyproject.toml:5`).

== Q1 — Problem and Users

=== Q1.1 Product identity
[ANSWERED] Mistral Vibe is "Mistral's open-source CLI coding assistant" — a
command-line conversational coding agent powered by Mistral models.
Evidence: `README.md:22`, `pyproject.toml:4` ("Minimal CLI coding agent by
Mistral"), package name `mistral-vibe` (`pyproject.toml:2`).

=== Q1.2 Primary users
[ANSWERED] Software developers working in a terminal on a code project. The
README addresses developers exploring, modifying, and interacting with their
codebase; the entrypoint targets a TTY (`vibe/cli/terminal_detect.py`).
Evidence: `README.md:24`, `docs/use_cases/UC-001-start-a-coding-session.md`.

=== Q1.3 Channels
[ANSWERED] Three delivery channels exist.
(a) Interactive TUI: `vibe` console script → `vibe.cli.entrypoint:main`
(`pyproject.toml`, project.scripts).
(b) Programmatic / headless mode: `vibe -p "<prompt>"`
(`vibe/core/programmatic.py`, `action.yml:54`).
(c) ACP server: `vibe-acp` console script → `vibe.acp.entrypoint:main`, the
Agent Client Protocol integration for editors/IDEs (`vibe/acp/`,
`docs/acp-setup.md`).
A GitHub Action wraps channel (b) (`action.yml`).

=== Q1.4 Why built
[ANSWERED — partial] Code shows *what*, not the strategic *why*. Vibe is an
open-source coding agent that binds Mistral models to a terminal workflow with a
tool sandbox. The strategic motivation (market positioning, adoption goals)
is not derivable. See `[OPEN — Q1.4-rationale]`.
Evidence (what): `README.md`, `vibe/core/agent_loop.py`.

[OPEN — Q1.4-rationale] Category: Product strategy. Ask: Product Owner. Why: the
business reason Mistral ships an open-source CLI agent is not in source.

=== Q1.5 Success metrics
[ANSWERED — partial] The product *measures* itself: telemetry (`enable_telemetry`
default `True`, `vibe/core/config/_settings.py:511`, `vibe/core/telemetry/`),
per-session cost and token stats (`AGENT_STATS` entity,
`vibe/core/agents/models.py`), and feedback bar
(`vibe/cli/textual_ui/widgets/feedback_bar.py`). The target *values* of any KPI
are not in code.

[OPEN — Q1.5-targets] Category: Product strategy. Ask: Product Owner. Why: no
acceptance target (DAU, retention, NPS) is encoded.

=== Q1.6 Segment priority
[OPEN — Q1.6] Category: Product strategy. Ask: Product Owner. Why: code does not
rank user segments. README warns Windows is unofficial and UNIX is "officially
supported and targeted" (`README.md:27`) — a *platform* priority, not a user
segment one; recorded here as the only code signal.

=== Q2.6 Cross-cutting business rules
[ANSWERED] Derivable rules:
BR-1 Tool calls require approval unless the agent profile or config bypasses it
(`vibe/core/config/_settings.py:510` `bypass_tool_permissions`,
`vibe/core/tools/permissions.py`).
BR-2 The `bash` tool denies interactive/editor commands and standalone
interpreters (`vibe/core/tools/builtins/bash.py:120-147`).
BR-3 Project-level config loads only from a trusted folder
(`vibe/core/trusted_folders.py`, UC-021).
BR-4 A session auto-compacts when context tokens exceed the model threshold
(`vibe/core/config/_settings.py:375` `auto_compact_threshold=200_000`).
BR-5 At most 50 scheduled loops per session (`vibe/core/loop.py:35`).
BR-6 LLM/HTTP requests retry on transient errors 408/409/425/429/500/502/503/504/529
(`vibe/core/utils/retry.py:11`).

== Q4 — Quality Goals (ISO/IEC 25010)

Quality is derived from measurable code behaviour. Each characteristic carries at
least one `[ANSWERED]` scenario with `file:line`. Only the ranking (Q4.9) is OPEN.

=== Q4.1 Functional Suitability
[ANSWERED] 12 built-in tools, 4 agent profiles, MCP + connector extensibility,
21 catalogued use cases. Functional completeness asserted by 228 test modules.
Evidence: `vibe/core/tools/builtins/`, `README.md` Built-in Agents,
`docs/use_cases/`.

=== Q4.2 Performance Efficiency
[ANSWERED] Scenario: LLM call budget — `api_timeout` default 720.0 s
(`_settings.py:520`); bash command default timeout 300 s
(`bash.py:225-226`); MCP startup 10.0 s, MCP tool call 60.0 s
(`_settings.py:234-240`); project-context scan 2.0 s
(`_settings.py:146`). Auto-compaction caps context at 200_000 tokens
(`_settings.py:375`). Async streaming throughout (`anyio`, `agent_loop.py`).

=== Q4.3 Compatibility
[ANSWERED] Five LLM backends behind one factory
(`vibe/core/llm/backend/factory.py`); MCP transports stdio/http/streamable-http
(`_settings.py:229-324`); ACP 0.9.0 for editor interop; OS keyring abstraction;
Windows + UNIX both run, UNIX targeted (`README.md:27`).

=== Q4.4 Usability
[ANSWERED] Onboarding flow (`vibe/setup/onboarding/`), autocompletion for `/`
and `@` (`vibe/cli/autocompletion/`), persistent input history
(`vibe/cli/history_manager.py`), snapshot tests pin TUI rendering
(`tests/snapshots/`), update notifications and "what's new"
(`vibe/cli/update_notifier/whats_new.py`).

=== Q4.5 Reliability
[ANSWERED] Scenario: transient LLM/HTTP failure — retry 3 tries, 0.5 s initial
delay, ×2 backoff, on status 408/409/425/429/500/502/503/504/529
(`utils/retry.py:11,17`). Non-retryable and rate-limit errors detected and
surfaced (`agent_loop.py:166-187`). Sessions persisted to disk for resume
(`vibe/core/session/resume_sessions.py`, UC-009). Session migration handles
format drift (`session_migration.py`).

=== Q4.6 Security
[ANSWERED] Scenarios:
(a) LLM-proposed tool call is gated by approval unless bypassed
(`tools/permissions.py`, `_settings.py:510`).
(b) `bash` denies interactive shells/editors and standalone interpreters
(`bash.py:120-147`); path commands checked against the working directory
(`bash.py:150+`, `tools/utils.is_path_within_workdir`).
(c) Project config loads only from trusted folders (`trusted_folders.py`).
(d) Credentials stored in OS keyring, never plaintext config (`auth/github.py`).
(e) Remote payloads use AES-256-GCM + RSA-OAEP-SHA256, RSA key ≥2048 bits,
ciphertext capped at 2 MiB, encrypted-key capped at 1 KiB
(`auth/crypto.py:13-17`).

=== Q4.7 Maintainability
[ANSWERED] Ports-and-adapters seams (`*/ports/`, `*/adapters/`); ruff + pyright
+ vulture + typos enforced via pre-commit (`.pre-commit-config.yaml`,
`pyproject.toml`); pylint limits — max 50 statements, 15 branches, 15 locals per
function (`pyproject.toml` tool.ruff.lint.pylint); 228 test modules; mandatory
`from __future__ import annotations` (`pyproject.toml`).

=== Q4.8 Portability
[ANSWERED] Python ≥3.12 only constraint (`pyproject.toml:6`); ships as wheel,
PyInstaller binary, and Nix flake (`pyinstaller/`, `flake.nix`); platform branches
isolated in `utils/platform.py` and `is_windows()` checks; `XDG`/home path
conventions in `vibe/core/paths/`.

=== Q4.9 Quality-goal priority ranking
[OPEN — Q4.9] Category: Quality goals. Ask: Architect / Product Owner. Why: code

== Q5 — Risks

=== Q5.1 Technical debt
[ANSWERED — partial] Derivable signals: `vulture` is configured, implying dead-
code vigilance (`pyproject.toml`); large modules exist (`agent_loop.py` >1300
lines). No explicit `TODO`/`FIXME` debt register found in the sampled core.
Full debt inventory needs the team.

[OPEN — Q5.1-register] Category: Technical debt. Ask: Developer. Why: code lacks
a debt log; intentional shortcuts are not annotated.

=== Q5.2 Security risks
[ANSWERED] Principal risk: an LLM proposes a destructive shell command and the
user approves it, or runs under `auto-approve` / `bypass_tool_permissions`
(`_settings.py:510`, README "Use with caution"). Secondary: a malicious project
folder's `AGENTS.md`/config is loaded if mistakenly trusted
(`trusted_folders.py`). MCP servers run as child processes with inherited env
(`_settings.py:319` env passthrough). See arc42 §8.1 STRIDE catalogue.

=== Q5.3 Operational risks
[ANSWERED — partial] LLM API outage or rate limit halts the agent; retry softens
transient cases only (`utils/retry.py`). 720 s `api_timeout` means a hung call
blocks the session up to 12 minutes (`_settings.py:520`). Telemetry is on by
default (`_settings.py:511`) — a privacy/operations concern. SLO/incident
process is not in code.

[OPEN — Q5.3-slo] Category: Operations. Ask: Operations. Why: no SLO, alerting,
or on-call process is encoded.

=== Q5.4 Dependency / supply-chain risks
[ANSWERED] `agent-client-protocol==0.9.0` and `mistralai==2.4.4` are exact pins
and explicitly excluded from `exclude-newer` auto-bump
(`pyproject.toml` tool.uv) — drift risk if upstream changes. `uv.lock`
(339 KB) pins the full tree. Cryptography, keyring, mcp, textual are direct
trust-sensitive dependencies. No SBOM or signature verification of installs
found.

[OPEN — Q5.4-sbom] Category: Supply chain. Ask: Operations. Why: code shows no
SBOM generation or dependency-CVE gate.

=== Q5.5 Scaling / performance risks
[ANSWERED] Single-user, single-process CLI — no horizontal scaling concern.
Risk is per-session: context growth is bounded by auto-compaction at 200_000
tokens (`_settings.py:375`); scheduled loops capped at 50
(`vibe/core/loop.py:35`); file indexer watches the project tree
(`vibe/core/autocompletion/file_indexer/watcher.py`) — large repos may stress
the watcher. Subagent delegation (`task` tool) fans out work in-process.
