== About this document

This is the *Phase 1* output of the Socratic Code-Theory Recovery workflow
(Phase 0.5 of the Semantic Anchors brownfield process). It recovers the mental
model of the *Mistral Vibe* bounded context — an open-source Python CLI coding
agent — through recursive question decomposition *before* any documentation is
synthesized.

* *Bounded context:* the whole `mistral-vibe` repository, treated as one
  context (the Vibe coding agent). Its four top-level modules — `vibe/core`,
  `vibe/cli`, `vibe/acp`, `vibe/setup` — could each be recursively decomposed
  further if a finer-grained context boundary is desired.
* *Decomposition anchors:* arc42 (Q3), Cockburn Use Cases (Q2),
  ISO/IEC 25010 (Q4), Nygard ADRs (Q3.9).
* *Leaf legend:*
** `[ANSWERED]` — derivable from code/config; evidence cited as `file:line`
   or `file::symbol`.
** `[OPEN]` — not derivable from code; tagged with a Category and an Ask role.
   See `OPEN_QUESTIONS.adoc`.

Stop rule: a branch stops decomposing when a question is answerable with a
single code reference or a single stakeholder fact.

== Q3 — What is the architecture of this bounded context?

Decomposed with the arc42 twelve-chapter anchor.

=== Q3.1 — Introduction and Goals
`[ANSWERED]` Purpose and feature set: `README.md:92-177`. Top-level quality
goal priorities are `[OPEN]` — see Q4.9.

=== Q3.2 — Constraints
`[ANSWERED]` Python ≥ 3.12 (`pyproject.toml:21`); managed exclusively with `uv`
(`AGENTS.md` "Commands"); UNIX officially targeted, Windows best-effort
(`README.md:24-25`); Apache-2.0 license (`LICENSE`, `pyproject.toml:14`);
coding-style constraints (`AGENTS.md` "Python style").

=== Q3.3 — Context and Scope
`[ANSWERED]` External dependencies: Mistral API (LLM, transcription, TTS,
telemetry); MCP servers; ACP clients; PyPI and GitHub (update checks —
`vibe/cli/update_notifier/`); OTel collector (`vibe/core/tracing.py`).

=== Q3.4 — Solution Strategy
`[ANSWERED]` (structure) Hexagonal-style ports/adapters (`_port.py` suffix,
`AGENTS.md`); Textual TUI; async (`anyio`/`asyncio`); layered configuration.
Rationale for these choices is `[OPEN]` — see Q3.9.

=== Q3.5 — Building Block View
`[ANSWERED]` Four top-level modules (`AGENTS.md:5`): `vibe/core` (engine: agent
loop, tools, LLM backends, config), `vibe/cli` (Textual TUI), `vibe/acp` (ACP
bridge), `vibe/setup` (first-run wizards).

=== Q3.6 — Runtime View
`[ANSWERED]` `AgentLoop` orchestrates turns (`vibe/core/agent_loop.py:221`),
running each turn through a middleware pipeline
(`vibe/core/agent_loop.py:281-282`, `vibe/core/middleware.py:218`). Deferred
heavy initialization runs on a background thread
(`vibe/core/agent_loop.py:241-246`, `:349-354`).

=== Q3.7 — Deployment View
`[ANSWERED]` Install via `uv tool install` / `pip` / `install.sh`
(`README.md:27-55`, `scripts/install.sh`); standalone PyInstaller binaries
(`pyinstaller/`, `vibe-acp.spec`, `distribution/`); GitHub composite Action
(`action.yml`). Two console entrypoints: `vibe`, `vibe-acp`
(`pyproject.toml:73-75`).

=== Q3.8 — Crosscutting Concepts
`[ANSWERED]` Logging (`vibe/core/logger.py`); telemetry
(`vibe/core/telemetry/`); OpenTelemetry tracing (`vibe/core/tracing.py`);
layered config merge with declarative merge strategies
(`vibe/core/config/schema.py`, `vibe/core/config/layer.py`); tool permission
model (`vibe/core/tools/permissions.py`); trust-folder gating
(`vibe/core/trusted_folders.py`); hooks (`vibe/core/hooks/`).
