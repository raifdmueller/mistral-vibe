[[Q2_3]]
=== Q2.3 What are the on-disk data formats?

[[Q2_3_1]]
==== Q2.3.1 Where does Vibe store state?

`[ANSWERED]` — All paths derive from `VIBE_HOME` (default `~/.vibe/`,
override via `VIBE_HOME` env var). Constants:
`vibe/core/paths/_vibe_home.py:24-39`. Files: `config.toml`, `.env`,
`trusted_folders.toml`, `cache.toml`, `vibehistory`, `logs/vibe.log`,
`logs/session/<session-dir>/{meta.json,messages.jsonl}`,
`plans/`, `agents/`, `prompts/`, `skills/`.

[[Q2_3_2]]
==== Q2.3.2 What is the session-on-disk format?

`[ANSWERED]` — Each session is a directory `session_{timestamp}_{short_id}`
containing `meta.json` (`vibe/core/session/session_logger.py` uses
`SessionMetadata` from `vibe/core/types.py:146-156`) and
`messages.jsonl` (one `LLMMessage` per line, see
`vibe/core/types.py:215`).

[[Q2_3_3]]
==== Q2.3.3 What is the skill format?

`[ANSWERED]` — Markdown file with YAML front-matter validated by
`SkillMetadata` (`vibe/core/skills/models.py:8-58`). Required: `name`
(kebab-case, ≤64 chars), `description` (≤1024 chars). Optional:
`license`, `compatibility`, `metadata`, `allowed-tools`,
`user-invocable`. See `docs/use_cases/UC-011-invoke-a-user-skill.md`.

[[Q2_3_4]]
==== Q2.3.4 What is the agent profile format?

`[ANSWERED]` — TOML file loaded by `AgentProfile.from_toml`
(`vibe/core/agents/models.py:88-99`). Accepted keys: `display_name`,
`description`, `safety`, `agent_type`, plus any field that overlays
`VibeConfig`.

=== Q3.2 Constraints

[[Q3_2_1]]
==== Q3.2.1 Language and runtime constraints

`[ANSWERED]` — Python 3.12+ only (`pyproject.toml:6,106,131`). Strict
type-checking with pyright (`pyproject.toml:115-122`). Lint and format
via Ruff (`pyproject.toml:124-160`).

[[Q3_2_2]]
==== Q3.2.2 Distribution constraints

`[ANSWERED]` — Two scripts: `vibe` and `vibe-acp`
(`pyproject.toml:75-77`). Distributed via PyPI and a one-line install
script (`README.md:30-46`). PyInstaller artifact (`vibe-acp.spec`)
exists for bundled binaries.

[[Q3_2_3]]
==== Q3.2.3 Coding constraints encoded in tooling

`[ANSWERED]` — `AGENTS.md` documents conventions: `uv` for all
commands; hexagonal-style ports via `_port.py` suffix; pydantic for
all external parsing; no relative imports (`AGENTS.md:42`); no inline
`# type: ignore` or `# noqa` (`AGENTS.md:44`); `StrEnum`/`IntEnum`
with `auto()`; `pathlib.Path`/`anyio.Path` instead of `os.path`;
`from __future__ import annotations` enforced by
`pyproject.toml:159` (`required-imports`).

[[Q3_2_4]]
==== Q3.2.4 Why these constraints?

`[OPEN]` — The constraints are encoded but their rationale (why pyright

[[Q3_4_1]]
==== Q3.4.1 What architectural style was chosen?

`[ANSWERED, partial]` — Hexagonal / ports-and-adapters is the
intended style (`AGENTS.md:23`, `_port.py` suffix convention). The
`vibe/cli/plan_offer/ports/` directory demonstrates it. However the
core engine (`vibe/core/agent_loop.py`, 1716 lines) is not strictly
hexagonal — backends and tools are wired by string identifiers rather
than injected ports.

* **[OPEN]** Q3.4.1.a — Is hexagonal the official target style for

** Category: design-rationale
** Ask role: Architect

[[Q3_4_2]]
==== Q3.4.2 How is multi-provider LLM support structured?

`[ANSWERED]` — `APIAdapter` protocol (`vibe/core/llm/backend/base.py:19`)
implemented by `MistralBackend`, `GenericBackend` (wrapping
`OpenAIAdapter`, `AnthropicAdapter`, `OpenAIResponsesAdapter`,
`VertexAnthropicAdapter`). Factory in
`vibe/core/llm/backend/factory.py`.

[[Q3_4_3]]
==== Q3.4.3 How is the conversation loop driven?

`[ANSWERED]` — `AgentLoop` (`vibe/core/agent_loop.py:221`) orchestrates
LLM request -> tool resolution -> tool execution -> result injection,
yielding `BaseEvent` instances to the UI. Middleware chain in
`vibe/core/middleware.py` decorates the loop.

`[OPEN]` — No ADR directory exists in the repository. Every notable

[[Q3_9_1]]
==== Q3.9.1 ADR candidates (inferred from code)

Each item below is an ADR-shaped question that has no answer in the
repository:

. **Choose Textual (8.2.4) as the TUI framework** — pinned at
  `pyproject.toml:53`. Alternatives: Rich-only, prompt_toolkit,
  custom curses, web UI. Rationale unrecorded.
. **Pin `mistralai==2.4.4` and `agent-client-protocol==0.9.0`** while
  `[tool.uv]` excludes upgrades within the last 7 days
  (`pyproject.toml:81-83`). Rationale and freshness policy
  unrecorded.
. **Hexagonal "_port.py" suffix convention** (`AGENTS.md:23`) —
  alternatives: layered, clean architecture, no formal structure.
. **Asyncio (rather than threads/trio) as the orchestration model**
  (`AGENTS.md:51`).
. **Pydantic for all external parsing** (`AGENTS.md:46`) —
  alternative: dataclasses + manual validation, attrs, msgspec.
. **Telemetry routed through the primary Mistral provider** even
  when other providers are active (`vibe/core/telemetry/send.py:50-60`)
  — alternative: a dedicated telemetry endpoint.
. **Plain Python script install via `curl | bash`** (`README.md:33`)
  — alternative: brew, deb/rpm, signed binaries only.
. **Agent profiles override `VibeConfig` rather than being a
  parallel schema** (`vibe/core/agents/models.py:46-72`,
  `_deep_merge`) — alternative: profile DSL separate from settings.
. **MCP "transport" discriminated union** with sibling classes
  (`vibe/core/config/_settings.py:307-339`) — chosen per
  `AGENTS.md:49`.
. **Session log format = JSONL + JSON metadata** rather than SQLite
  or a single document — see `vibe/core/session/session_logger.py`.

Each of the above is a separate **[OPEN]** ADR (design-rationale,
Architect).

[[Q4_5]]
=== Q4.5 Reliability

[[Q4_5_1]]
==== Q4.5.1 Crash recovery

`[ANSWERED, partial]` — Session log is append-only JSONL, so a crash
mid-turn leaves a recoverable transcript that `--resume` can
re-open (`vibe/core/session/session_loader.py::_is_valid_session`
gates on minimum validity). Partial filesystem side-effects from a
crashed tool are not rolled back.

[[Q4_5_2]]
==== Q4.5.2 Network-failure tolerance

`[ANSWERED, partial]` — Rate-limit detection
(`vibe/core/agent_loop.py:166-178`) and a single retry-after policy
are present. A general retry/circuit-breaker policy is **[OPEN]**
(see Q3.8.4.a).

[[Q4_6]]
=== Q4.6 Security

[[Q4_6_1]]

(`vibe/core/config/_settings.py:57-66`); keyring dependency
present (`pyproject.toml:42`) for OS-level secret stores.

[[Q4_6_2]]

`[ANSWERED]` — None. Tools run with the user's full privileges. The

and not surfaced as a per-call consent. The `data_retention.py`
message points users to Mistral's settings page.
